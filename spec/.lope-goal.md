# Lope Negotiate Goal — Makakoo OS Architecture Validation

## Mission

We are building **Makakoo OS**, a parasite-class AI operating system. It runs as
userland on macOS, Linux, and Windows today. It must also be designed so the
Rust kernel can compile and run on Redox OS in a future sprint without any
rewrite. The parasite metaphor is literal — Makakoo infects existing AI CLIs
(Claude Code, Gemini CLI, Codex, Cursor, Vibe, OpenCode, Qwen) and IDEs by
writing a Bootstrap Block into their global instructions and symlinking their
memory and skills directories to shared Makakoo state. One mind, many bodies.

The user's hard requirement: **no rebuild later**. Every load-bearing decision
must be correct on paper before any migration. We lock them now, review them,
and don't ship Phase B until every decision is green.

## The 10 load-bearing decisions to validate

1. **Rust kernel, 4-target cross-OS from day one.** Cargo workspace, compiles to
   macOS (x86_64 + aarch64), Linux (x86_64 + aarch64), Windows (x86_64), and
   Redox (x86_64). Target-gated code lives in `crates/platform/<os>/`, never in
   `crates/core/` or `crates/cli/`.

2. **Python is a plugin, not a dependency.** The Rust kernel boots with zero
   Python. Python enters only via plugins that declare `language = "python"` in
   their manifest. A fresh install with no Python still runs the kernel.

3. **Plugins are directories with `plugin.toml` manifests.** The manifest is the
   only contract. No code discovery by import reflection. Walk filesystem,
   parse TOML, load. The manifest format is Redox-recipe-inspired:
   `[plugin]`, `[source]`, `[abi]`, `[depends]`, `[install]`, `[entrypoint]`,
   `[capabilities]`, `[sancho]`, `[mcp]`, `[infect]`, `[state]`, `[test]`.

4. **State is filesystem-first.** Brain and auto-memory are flat markdown files
   (Logseq-compat). Superbrain.db is a derived cache that can be regenerated
   from the markdown. Plugins write only to their own state dir enforced via
   capability tokens.

5. **Capabilities declared in manifest, enforced by kernel.** Every plugin lists
   exactly what verbs it uses (`brain/read`, `llm/chat:<model>`,
   `net/http:<url-glob>`, `state/plugin`, `exec/binary:<allowlist>`, etc.).
   Daemon issues a signed token at plugin start. Makakoo helper libs check the
   token on every call. This is an honesty boundary, not a sandbox — we accept
   a plugin that shells out to curl can bypass it.

6. **ABIs are semver-versioned.** Plugins declare e.g. `abi = "agent/^1.0"`.
   Kernel refuses incompatible plugins at load time. Breaking ABI changes bump
   major, additions bump minor, clarifications bump patch.

7. **Distros are opinionated plugin lists.** A distro file
   (`distros/trader.toml`) is a declarative kernel version + plugin list + default
   config. `makakoo distro install <name>` batch installs.

8. **The CLI is the only user-facing surface.** No TUI, no GUI, no web UI. If
   we ever add a TUI it ships as a plugin, not as kernel code.

9. **One umbrella install command.** `makakoo install` does: plugin install
   core distro + daemon install + infect auto-detect + symlink shared state.
   Three granular commands (`setup`, `infect --refresh`, `plugin install
   <name>`) exist for finer control but the umbrella is the blessed path.

10. **Cross-OS from day one.** Every decision checked against all 4 targets. If
    it works on macOS but can't be made to work on Windows without a
    Windows-only fork, we don't ship it.

## What to validate

For each of the 10 decisions, independently assess:

- **Correctness.** Is this decision internally consistent? Does it conflict
  with another decision? Does it assume a capability that doesn't exist in one
  of the 4 target OSes?

- **Future-proofing.** Does this decision survive the Redox port? Does it
  survive scaling to 100+ plugins? Does it survive a user who wants to swap a
  core plugin for a community alternative?

- **Rebuild risk.** If we discover after 6 months that this decision was
  wrong, does fixing it require a rewrite, or is it an additive change?
  Specifically identify any decision that would force a full rebuild if
  reversed.

- **Honest gotchas.** What is this decision silently papering over? Where does
  the abstraction leak? What edge case is going to bite us?

## What to propose

Any concrete fix needed before Phase B code work starts. Be specific —
"consider adding foo to bar" is useless. "Replace decision N with X because Y"
is useful.

Also flag any missing decision that should be part of the 10 but isn't.

## Format

Return the sprint doc as a markdown file with these sections:

- **Verdict** (one of: PASS, NEEDS_FIX, FAIL) — based on whether any decision
  has unresolvable rebuild risk
- **Per-decision scorecard** — for each of decisions 1-10, one paragraph
- **Fixes required before Phase B** — numbered list, ranked by severity
- **Missing decisions** — anything the user should add to the 10
- **Confidence** — 0.0 to 1.0

Keep it under 800 words total. Dense.
