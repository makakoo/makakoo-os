## The headline

The spec is structurally sound and the parasite/shared-state model is
internally coherent, BUT **4 locked decisions have leaks that would force
a rebuild if not addressed before Phase B**. The 10-12 day sprint estimate
is also optimistic by 40-60%. Realistic total: 17-22 days (3 weeks, not 2).

## Critical issues ranked by rebuild risk

### 1. Async runtime is not named (must become decision 11)
Tokio vs async-std vs smol vs sync-only is the single most rebuild-forcing
call in the entire kernel. Tokio on Redox is a known sore spot (mio backend
gaps on stable rustc). Spec says "Redox-compat" but doesn't name the runtime.
**Fix:** lock the async runtime choice in section 4 as a new decision.

### 2. Windows shared state is hand-waved
Decision 4 says "filesystem-first" but the install flow concedes Windows uses
"copy-sync fallback." Copy-sync is a different system with its own consistency
model — two writers will produce divergent state. Spec also claims Windows is
"first-class."
**Fix:** pick one. Either REQUIRE Windows Developer Mode symlinks as an
install precondition (Win10 1703+ available) or accept Windows is a degraded
tier. Don't ship copy-sync.

### 3. Superbrain ownership is double-claimed
`crates/core` names Superbrain as a kernel responsibility (section 5); but
`plugins-core/superbrain-py/` lists it as a plugin. Pick one.
- If core: it has a Python dependency via embeddings, violating decision 2.
- If plugin: the `brain_search` MCP tool cannot be a kernel-resident tool.
**Fix:** move Superbrain fully to a plugin. Keep only a lightweight
`brain_fts_search` in core that walks the markdown files directly.

### 4. `rusqlite` with `bundled` feature violates "no C in core"
Section 9.5 lists bundled C SQLite inside the kernel. Decision 1 says core
stays pure Rust. Contradiction.
**Fix:** either (a) move SQLite to a plugin (bigger refactor), or (b)
acknowledge core has one C dep and narrow decision 1 to "no OS-specific
code." I pick (b) — SQLite is portable C, not OS-specific.

### 5. Plugin install scripts are shell-shaped
Manifest uses `[install].script = """...sh-style..."""` which won't run on
Windows PowerShell.
**Fix:** require every plugin to ship `install.sh` + `install.ps1` pair,
referenced by manifest as `[install].unix` + `[install].windows`. Or declare
a portable install DSL (over-engineering — pick pair).

### 6. Six ABIs frozen at v1.0 in Phase A with zero implementation experience
Real ABIs require at least one consumer round-trip before locking.
**Fix:** ship v0.x ABIs through Phase D. Promote to v1.0 only after Phase E
dogfooding. Update section 4 decision 6 to "ABIs are semver-versioned; v0.x
during migration, v1.0 after Phase E validation."

### 7. Capability token model (HMAC) is wrong tool for honesty boundary
Three sub-issues:
- **HMAC implies threat protection you explicitly disclaim.** If it's not a
  security boundary, a UUID per process in a daemon table works identically
  without secret management.
- **Token leaks to child subprocesses via env var.** Python plugin reads
  `MAKAKOO_CAPABILITY_TOKEN`, calls `subprocess.run()`, child inherits env,
  token leaks everywhere. Week-one bug.
- **No token revocation on crash, no transactional writes specified.**
**Fix (stronger alternative):** per-plugin Unix domain socket (named pipe on
Windows), kernel verifies PID on accept. No token at all. Maps cleanly to
Redox channel schemes (`chan:`). Section 9.6 already says capabilities map
to schemes — so the enforcement primitive should BE a scheme from day one.

### 8. Event bus semantics unspecified
Section 3 lists "EventBus" in `crates/core` but section 4 doesn't lock its
semantics. In-process pub/sub? SQLite-backed queue? Unix-socket fanout? Each
picks different OS primitives and different Redox-compat profiles.
**Fix:** name it. My pick: in-process `tokio::sync::broadcast` for live
subscribers + a SQLite journal for persistence + replay. Both cross-OS. If we
go sync-only (see issue 1), use `crossbeam::channel` instead.

### 9. Missing decisions that should be in locked section 4
Promote from section 12 (open questions) to section 4 (locked):
- **12.3** — plugin process model (subprocess vs dylib). Pick subprocess. Lock.
- **12.6** — Bootstrap Block rendering (on-read vs on-event). Pick on-event
  with cache. Lock.
- **12.7** — cross-OS path handling via `$MAKAKOO_HOME`. Lock as hard constraint.

Add as new locked decisions:
- **11** — async runtime choice (see issue 1)
- **12** — Windows symlink policy (see issue 2)
- **13** — embedding model + dimensionality lock (past `auto-memory` migration
  bit you via Qdrant/pgvector dimensionality mismatch per memory
  `project_embedding_migration`). Lock the embedding contract.
- **14** — Bootstrap Block merge semantics when two plugins contribute
  conflicting fragments. Current spec has no merge policy.
- **15** — event bus semantics (see issue 8)

Final locked count: **15 decisions**, not 10.

## Cross-OS technical gotchas

- **SQLite FTS5 + WAL mode** uses mmap aggressively. Redox mmap support is
  partial. WAL on Redox is a known unknown. Test in Phase H or budget for a
  pure-Rust SQLite fallback (which doesn't exist yet).
- **`std::fs` symlink semantics** differ across Linux/macOS/Windows/Redox.
  `fs::symlink_metadata` vs `metadata` will bite.
- **MCP gateway stdio.** `std::io::stdin().lock()` is fine cross-OS. But
  `tokio::io::stdin` on Windows is blocking in a way that breaks the event
  loop. Pick the runtime before Phase C.
- **launchd KeepAlive vs systemd Restart=** semantics differ on crash
  backoff. Not addressed in section 6.
- **Windows Task Scheduler non-admin** can't run at boot before login.
  Daemon is login-scoped on Windows. That's actually consistent with macOS
  LaunchAgent, but not documented.
